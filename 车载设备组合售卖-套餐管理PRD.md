# 车载设备组合售卖场景套餐管理功能 PRD

## 1. 文档目标

本文档用于定义面向车载设备组合售卖场景的套餐管理能力，覆盖套餐配置、版本管理、售卖下单、履约交付、售后变更与年费续费等核心流程，为产品、设计、研发、测试和运营提供统一实现口径。

核心目标如下：

- 支持将定位终端、盲区摄像头等硬件按套餐组合售卖
- 支持硬件一次性收费与按购买日起算的年管理费/服务费并存
- 支持监控服务、上门安装等选配或非必选服务
- 支持增购配件、补装、迁移、退货协商、续费等标准售后流程
- 支持套餐内容与价格频繁调整下的版本化发布与历史追溯

## 2. 业务范围

### 2.1 管理端范围

- 套餐配置：组合规则、必选条件、价格、年费、服务项、渠道可见性、客户类型适用范围
- 套餐版本管理：版本创建、复制、编辑、校验、发布、生效、停用、归档
- 售卖支持：线上售卖配置、管理端线下开单、指定版本销售
- 售后管理：增购、补装、迁移、退货协商、退款留痕
- 年费续费管理：标准续费、协商价续费、到期提醒、服务有效期延长

### 2.2 用户端范围

- 线上浏览并购买套餐
- 选择监控服务、安装服务等选配项
- 查看订单、服务状态、续费状态
- 完成车辆与设备绑定相关动作

### 2.3 不在本期明确范围

- 年费退款的自动按天折算规则
- 多设备绑定一车或一车绑定多设备的复杂模型
- 并行售卖策略的最终规则细节
- 补装与迁移的完整服务时效和费用策略

## 3. 核心概念定义

- **套餐 Package**：对外售卖的稳定商品主体，拥有稳定 `package_id`。
- **套餐版本 Package Version**：套餐在某一时点的可售卖快照，包含内容、规则、价格、服务项等信息，拥有独立 `package_version_id`。
- **套餐项 Package Item**：某个套餐版本下的组成项，分为必选设备项、选配服务项、加购配件项。
- **设备 SKU**：可售卖硬件商品，交付时需采集唯一标识，例如 `SN`、`IMEI`。
- **车辆绑定**：设备与车辆的绑定关系。本期约束为一辆车只能绑定一个设备。
- **年费**：按购买日起算的年度管理费/服务费，可与硬件费用拆分支付、拆分开票。

## 4. 业务规则

### 4.1 套餐与套餐项

- 一个套餐可包含多个设备 SKU。
- 每个设备 SKU 支持配置数量，且支持最小值、最大值或固定数量。
- 套餐项需支持必选、选配、加购三种类型。
- 套餐项之间需支持依赖、互斥、必选条件等规则。

### 4.2 设备与绑定

- 硬件设备在出库或安装阶段必须采集唯一标识，如 `SN`、`IMEI`。
- 一辆车只能绑定一个设备。
- 若发生补装、迁移、替换等场景，需通过标准售后流程执行解绑与重绑。

### 4.3 计费与支付

- 套餐订单由硬件费和年费组成。
- 硬件费为一次性收费。
- 年费按购买日起算年度周期。
- 硬件费与年费支持拆分支付。
- 硬件费与年费支持拆分开票。

### 4.4 服务项规则

- 监控服务为订单级选配项，不是默认必选。
- 上门安装为非必选服务，可由客户自主选择。
- 若订单包含监控服务，则履约后需按开通条件开通服务。
- 若订单包含安装服务，则需生成安装工单并跟踪履约结果。

### 4.5 客户与销售渠道

- 支持个人客户与企业/车队客户。
- 支持线上自助下单与管理端线下开单。
- 线下开单允许在权限控制下选择指定套餐版本。

### 4.6 售后与退款

- 支持售后增购配件，并生成追加订单。
- 增购配件如影响年费，年费金额由人工协商后记录结果。
- 补装、迁移属于标准售后流程。
- 硬件退货后的年费退款采用协商处理，不做系统自动按天折算。

## 5. 产品设计原则

- **稳定售卖主体**：对外始终展示套餐，内部通过版本承载变化。
- **历史可追溯**：订单必须绑定下单时版本，后续改版不影响历史履约。
- **配置驱动**：规则、价格、服务项尽量通过配置实现，减少频繁改码。
- **操作留痕**：发布、停用、线下指定版本销售、协商价格与退款均需可审计。
- **售卖履约一体化**：套餐配置需兼顾售卖、交付、售后与续费场景。

## 6. 功能方案

### 6.1 套餐管理

管理端需支持：

- 创建套餐基础主体
- 维护套餐名称、介绍、渠道可见性、适用客户类型、整体上架状态
- 查看套餐下全部历史版本
- 控制套餐整体上架/下架，不直接影响历史订单履约

### 6.2 套餐版本管理

#### 6.2.1 版本结构

版本需至少包含：

- 基础信息：`package_version_id`、`package_id`、`version_no`
- 生命周期状态：`draft`、`published`、`active`、`inactive`、`archived`
- 生效区间：`effective_from`、`effective_to`
- 变更说明
- 价格配置：硬件价、年费价
- 规则配置：必选、选配、加购规则
- 套餐项明细

#### 6.2.2 生命周期规则

- 已生效版本不可直接编辑。
- 对已生效版本的调整必须通过复制创建新草稿版本完成。
- 草稿版本可编辑、校验、删除。
- 发布后可立即生效，也可定时生效。
- 停用只影响后续新单，不影响历史订单履约和售后追溯。

#### 6.2.3 标准操作流程

`复制当前版本 -> 形成草稿 -> 编辑 -> 校验 -> 发布 -> 立即/定时生效 -> 后续停用/归档`

#### 6.2.4 核心校验规则

- 版本号唯一且按套餐维度递增
- 同一套餐的生效时间区间不可非法重叠
- 必选规则必须可被满足，不能出现不可下单组合
- 依赖与互斥规则不能冲突
- 价格字段必须完整，不能发布缺失核心价格的版本

### 6.3 下单选版

#### 6.3.1 默认规则

- 线上下单时，系统按下单时间匹配当前有效版本。
- 新订单创建时必须写入 `package_version_id`。
- 订单需保存版本快照关键字段，至少包括套餐名、价格、套餐项、服务项和关键规则。

#### 6.3.2 线下开单规则

- 管理端允许销售选择指定版本下单。
- 指定历史版本下单需校验权限。
- 系统需记录选择原因、操作人、操作时间和原始默认版本。

### 6.4 订单与计费

订单层需支持：

- 套餐主商品下单
- 选配服务选择
- 生成硬件订单行与年费订单行
- 硬件费、年费拆分支付
- 硬件费、年费拆分开票
- 订单状态、支付状态、履约状态独立跟踪

建议订单模型拆分为：

- 套餐主订单
- 费用明细行
- 履约明细行
- 服务有效期记录

### 6.5 履约交付

#### 6.5.1 硬件履约

- 出库/发货时记录设备 SKU 与唯一标识
- 交付完成后触发车辆绑定
- 绑定时校验一车一设备规则

#### 6.5.2 服务履约

- 监控服务：满足开通条件后创建服务开通记录
- 安装服务：生成安装工单，跟踪预约、上门、完成、失败等状态

### 6.6 售后管理

#### 6.6.1 增购配件

- 支持基于原订单或原车辆发起配件增购
- 生成独立追加订单
- 如影响年费，记录协商价、生效时间、审批人和备注

#### 6.6.2 补装与迁移

- 补装支持对原套餐已购能力的补履约
- 迁移支持原设备解绑、新设备或新车辆重绑
- 迁移需记录原绑定关系、新绑定关系和服务延续规则

#### 6.6.3 退货退款

- 发起售后申请
- 经协商审批后分别处理硬件退款与年费退款
- 系统记录协商结果、审批链路、执行结果

### 6.7 年费续费管理

- 支持基于服务到期时间发起续费提醒
- 支持续费订单创建与支付
- 支持标准价续费与协商价续费
- 续费成功后延长服务有效期
- 续费记录需关联原套餐版本和当前续费价格依据

## 7. 关键流程

### 7.1 配置与发布流程

1. 创建套餐主体
2. 创建 V1 草稿版本
3. 配置套餐项、价格、年费、规则
4. 完成校验并发布
5. 立即或定时生效
6. 后续如需调整，从当前版本复制生成 V2

### 7.2 售卖与计费流程

1. 用户选择套餐
2. 系统匹配可售版本
3. 用户选择监控服务、安装服务等选配项
4. 系统生成订单及费用行
5. 完成支付与开票处理

### 7.3 履约流程

1. 硬件出库/发货
2. 采集 SN/IMEI
3. 绑定车辆
4. 按订单内容开通监控服务或生成安装工单

### 7.4 售后与续费流程

1. 发起增购、补装、迁移或退款申请
2. 审批与协商
3. 生成追加订单或执行变更
4. 更新服务状态与有效期
5. 到期前触发续费提醒并支持续费下单

## 9. 状态设计建议

### 9.1 套餐版本状态

- `draft`：草稿
- `published`：已发布待生效
- `active`：当前生效
- `inactive`：已停用
- `archived`：已归档

### 9.2 订单相关状态

- 订单状态：待支付、部分支付、已支付、履约中、已完成、已取消
- 履约状态：待出库、待发货、待安装、服务待开通、履约完成、履约异常
- 售后状态：待审批、处理中、已完成、已拒绝
- 续费状态：待提醒、待支付、已续费、已过期

## 10. 权限与审计要求

- 套餐版本发布、生效、停用需权限控制
- 线下开单选择指定版本需权限控制
- 协商价续费、年费协商退款需审批或授权
- 所有关键操作需记录操作人、时间、前后值、原因说明

## 11. 非功能要求

- 版本切换必须保证历史订单数据可读、可查、可履约
- 关键规则校验需在配置阶段前置拦截
- 订单、履约、售后、续费之间需保持链路可追踪
- 价格与规则变更需支持审计查询

## 12. 待确认事项

- 新旧版本是否允许长期并行售卖，若允许，控制维度是时间、渠道、客户类型还是销售组织
- 线下销售选择历史版本的权限边界和审批要求
- 增购配件影响年费时，采用覆盖式协商价还是增量式协商价
- 补装与迁移是否产生额外费用，以及由谁发起和审批
- 企业客户是否需要按车队维度统一续费与开票
- 退货后服务关闭时点如何定义

## 13. 验收要点

- 能创建套餐主体并维护多个历史版本
- 已生效版本不可直接编辑，只能复制出新版本
- 新订单能准确绑定下单时版本
- 历史订单在版本变更后仍可查看原快照并继续履约
- 硬件费与年费可拆分支付、拆分开票
- 监控服务可选，上门安装非必选
- 支持增购配件、补装、迁移、退货协商、续费
- 一车一设备绑定规则可被系统校验并拦截异常
